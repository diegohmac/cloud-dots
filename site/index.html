01–03How it works

01 / BAKE

One CLI pass, offline

Point the CLI at an FBX scan. It scatters millions of candidate points across the surface and keeps them where the texture is bright, so the stipple comes from the data itself. An optional blue‑noise pass packs the dots evenly, like bubbles. 

# 48 MB scan in, ~10 MB .pfc out
npx cloud-dots-bake scan.fbx -o face.pfc --poisson

02 / LOAD

A single binary

Everything ships in one .pfc file: positions, brightness and surface normals in a compact header‑plus‑floats layout. No mesh, no texture, no parsing cost at runtime. 

import { loadCloud } from 'cloud-dots'

const cloud = await loadCloud('/face.pfc', (p) => bar.set(p))

03 / RENDER

Two ways in

Drop the turnkey viewer on any element, or take the raw THREE.Points object into a scene you already have, React Three Fiber included. 

import { ParticleFaceViewer } from 'cloud-dots'

const viewer = new ParticleFaceViewer(el, { src: '/face.pfc' })
viewer.pulse()

+What you get

  • Reads as a solid

    Baked normals fade out back‑facing dots, so the cloud never looks like a see‑through shell, and they drive soft front lighting for free.

  • Fake depth of field

    Dots away from the focal plane grow, soften and dim. One uniform moves the sharp plane wherever you want it.

  • Assemble entrance

    Every dot eases in from its own scattered position with per‑dot stagger. Replay it whenever you like.

  • Alive by default

    Each dot floats on its own little orbit, and pulse() sends a ripple through the whole cloud.

  • Uniform thinning

    Points are stored in random order, so the density knob thins the cloud evenly from 100% down to 5%.

  • No build step

    Plain ESM source, three.js as a peer dependency, zero other runtime deps. Works with Vite, Next, or a bare import map.
